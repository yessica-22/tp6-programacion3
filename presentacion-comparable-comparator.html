Ordenamiento de objetos:
Comparable & Comparator

Polimorfismo en acción

Programación III — Paradigma Orientado a Objetos Universidad Nacional de La Rioja 2026

El problema

Tenemos una lista de nuestros propios objetos y queremos ordenarla:

List<Estudiante> lista = new ArrayList<>();
lista.add(new Estudiante("LU-001", "Martín", 8.5, 22, 18));
lista.add(new Estudiante("LU-002", "Valeria", 7.2, 20, 15));

Collections.sort(lista);  // ❌ No compila
The method sort(List<T>) in the type Collections is not applicable for the arguments (ArrayList<Estudiante>)
¿Cómo sabe Java cuál objeto va primero
si Estudiante es una clase que inventamos nosotros?

La idea clave

Comparar es un comportamiento.
En POO, los comportamientos se modelan con interfaces.

Comparable<T> "Me comparo con otro T"
⟷
Comparator<T> "Comparo dos objetos T"

Esto es polimorfismo

La misma operación (sort()) produce distinto comportamiento
según qué implementación de la interfaz recibe.

Comparable<T>

El contrato de compareTo(T otro)

  • Negativo → this va antes que otro
  • Cero → son equivalentes según ese criterio
  • Positivo → this va después que otro

Implementación en Estudiante

Estudiante.java
public class Estudiante
        implements Comparable<Estudiante> {

    private String legajo;
    private String nombre;
    private double promedio;
    private int edad;
    private int materiasAprobadas;

    @Override
    public int compareTo(Estudiante otro) {
        return Double
            .compare(otro.promedio, this.promedio);
    }
}

Orden natural: promedio descendente. Siempre usar Double.compare(), nunca resta.

Orden natural

Muchas clases del JDK ya implementan Comparable:

  • String → orden lexicográfico
  • Integer, Double, BigDecimal → orden numérico
  • LocalDate, LocalDateTime → orden cronológico

La gran limitación

Comparable define un solo criterio
para toda la vida de la clase.

  • No podés cambiar el criterio sin recompilar
  • No podés tener dos órdenes naturales a la vez
  • ¿Qué hacés si no podés tocar la clase?
Un solo criterio no alcanza

Comparator<T>

Estrategia externa a la clase de dominio.
Estudiante no se modifica. Se define con una lambda o un method reference.

Con lambda explícita

(a, b) -> Integer.compare(a.getEdad(), b.getEdad())

Con Comparator.comparing()

Comparator.comparing(Estudiante::getEdad)
Comparator.comparing(Estudiante::getNombre)
Comparator.comparing(Estudiante::getPromedio)

Así se usa

lista.sort(Comparator.comparing(Estudiante::getEdad));
lista.sort((a, b) -> Integer.compare(a.materiasAprobadas, b.materiasAprobadas));
Comparator.comparing() recibe una función extractora
(qué atributo comparar) y devuelve el comparator listo para usar.
Podés crear infinitos criterios sin tocar el dominio.

Encadenamiento y orden inverso

Desempate con thenComparing()

Comparator<Estudiante> criterio =
    Comparator
        .comparing(Estudiante::getPromedio)
        .reversed()
        .thenComparing(Estudiante::getNombre);

Orden inverso con reversed()

// Ascendente por promedio (menor a mayor)
Comparator.comparing(Estudiante::getPromedio)

// Descendente por promedio (mayor a menor)
Comparator
    .comparing(Estudiante::getPromedio)
    .reversed()
thenComparing() solo se ejecuta si hay empate
en el criterio anterior. Ideal para tie-breakers.

Comparable vs Comparator

Comparable<T> Comparator<T>
Paquete java.lang java.util
Método compareTo(T o) compare(T a, T b)
Criterios posibles 1 solo (el natural) Infinitos
¿Modifica la clase? Sí No
Acoplamiento Alto Bajo
¿Cuándo? Hay un orden "obvio"
para ese tipo		 Múltiples criterios,
clase de terceros,
no queremos tocar el dominio
¿Vale la pena modificar compareTo()
cada vez que aparece un nuevo requisito de orden?

⚠️ El anti-patrón de la resta

❌ Nunca uses la resta para comparar números 
// ❌ PELIGROSO: parece tentador, pero falla
(a, b) -> a.getEdad() - b.getEdad()

¿Por qué falla?

Overflow de enteros:

Integer.MAX_VALUE - (-1)
    = -2147483648  // OVERFLOW → resultado negativo

Le estás diciendo a Java que
2147483647 < -1, lo cual es falso.

Esto rompe el contrato de compare:
transitividad, simetría y consistencia.

La solución correcta

Para int

// ✅ CORRECTO
(a, b) -> Integer.compare(a.getEdad(), b.getEdad())

Para double

// ✅ CORRECTO
(a, b) -> Double.compare(a.getPromedio(), b.getPromedio())

Para long

// ✅ CORRECTO
(a, b) -> Long.compare(a.getValor(), b.getValor())
Integer.compare(), Double.compare() y Long.compare() manejan todos los casos, incluyendo MAX_VALUE y MIN_VALUE.

Hacia Spring Boot

Controller GET /api/estudiantes ?sortBy=edad&order=desc
⬇
Service Map<String, Comparator> .get("edad").reversed()
⬇
Comparator comparing(Estudiante::getEdad)
⬇
JSON ordenado

El controller recibe sortBy y order,
el service elige el comparator del mapa = patrón Strategy.

Request y respuesta

// GET /api/estudiantes?sortBy=promedio&order=desc

[
  { "legajo": "LU-004", "nombre": "Camila",  "promedio": 9.1 },
  { "legajo": "LU-005", "nombre": "Lucas",    "promedio": 9.1 },
  { "legajo": "LU-008", "nombre": "Florencia", "promedio": 8.9 },
  { "legajo": "LU-001", "nombre": "Martín",   "promedio": 8.5 },
  ...
]

El @RestController recibe los query params y delega
en el @Service, que elige el comparator del mapa.

El controller no contiene lógica de selección de criterios.
Toda esa responsabilidad vive en el service = Strategy.

Manejo de errores

Si el cliente manda un sortBy que no existe:

// GET /api/estudiantes?sortBy=colorFavorito
// → HTTP 400 Bad Request
{
  "error": "Criterio inválido",
  "recibido": "colorFavorito",
  "validos": [
    "promedio", "edad",
    "nombre", "materiasAprobadas",
    "legajo"
  ]
}

El service lanza una excepción si la clave no está en el mapa.
El controller la convierte en ResponseEntity.badRequest().

No usamos switch ni if-else para elegir el comparator.
Usamos un Map<String, Comparator<Estudiante>>.
